# Scalpel 1.0.0-alpha.1

This prerelease packages the current Windows-hardened Scalpel baseline for wider testing. It is not the final `1.0.0` safety claim.

## Highlights

- Installable `scalpel` stdio MCP command with Codex configuration examples.
- Precise text mutation tools with dry runs, optimistic preconditions, path locks, commit-time revalidation, and post-commit verification.
- Metadata-only transaction records and startup recovery for text writes and moves.
- Killed-process fault injection across transaction, temp-write, rename, journal, and recovery cleanup stages.
- Workspace confinement with hidden-path and symlink protections.
- Expanded public-corpus stress across Express, Lodash, TypeScript, Kubernetes, and LLVM.
- Structured MCP success/error payloads, read-only resources, and package/MCP smoke coverage.

## Verification

- `pnpm validate`: 68 tests passed, MCP smoke 33/33, package smoke passed.
- Expanded corpus hardening: 40/40 required checks passed.
- Crash fault matrix: 13/13 required checks passed.
- Combined race/crash baseline: 42/42 required checks passed before the expanded corpus run.

The GitHub prerelease includes the corresponding machine-readable and Markdown hardening reports as assets.

## Known Gaps Before 1.0.0

- Streaming mutation for files larger than `maxReadBytes` is not implemented.
- Unix-like crash, fsync, and filesystem persistence evidence is not yet available; WSL was unavailable on the release machine.
- Parent-directory sync remains best-effort and platform-dependent.
- Cross-device move semantics and recovery are not implemented.
- Malicious same-user changes after a successful response are outside the current race guarantee.
- A crash while holding a lock can require the configured stale-lock interval before recovery.

Use this prerelease for evaluation and local development, not as a claim of universally crash-safe storage semantics.
